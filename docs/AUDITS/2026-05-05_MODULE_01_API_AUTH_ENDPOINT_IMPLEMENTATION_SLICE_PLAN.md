## Proposed Implementation Order

### Slice 1A — Auth Service Skeleton
Create internal service/module for:
- email normalization
- user lookup
- user status check
- terminal lookup
- role loading
- permission resolution placeholder
- menu_schema generation placeholder
- consistent response envelope

No password/session writes until helpers are ready.

### Slice 1B — Password Verification Helper
Implement password verification against `module01_user_auth`.

Requirements:
- never log password
- never log password_hash
- support `password_algorithm` values:
  - ARGON2ID
  - BCRYPT
- if exact hashing library is not present, block implementation and report dependency need
- no plaintext password storage

### Slice 1C — Session Token Helper
Implement opaque session token support.

Requirements:
- generate high-entropy token using CSPRNG
- recommended: 32-byte random token encoded as hex/base64url
- store only `session_token_hash`
- never log raw token
- bind session to `terminal_id`
- set `expires_at` according to 12-hour TTL
- write `module01_user_sessions` row
- return raw token only once in login response

### Slice 1D — Login Endpoint
Implement POST `/api/v1/auth/login`.

Flow:
1. validate request body
2. normalize email
3. lookup user
4. generic invalid credentials for public failure
5. verify ACTIVE status
6. verify auth record exists
7. check `locked_until`
8. verify password
9. validate terminal by `spreadsheet_id`
10. load active roles
11. resolve permissions/menu_schema
12. create session
13. write LOGIN_SUCCESS audit
14. return session/user/roles/permissions/menu_schema

Failure handling:
- LOGIN_FAILED audit where possible
- no enumeration leak
- no secrets in logs

### Slice 1E — Refresh Menu Endpoint
Implement POST `/api/v1/auth/refresh_menu`.

Flow:
1. read Authorization Bearer token
2. hash token and find session
3. validate `expires_at`/`revoked_at`
4. validate `X-EDS-Terminal-ID` matches session terminal
5. if terminal mismatch:
   - revoke session immediately
   - return AUTH_SESSION_INVALID
   - audit SESSION_TERMINAL_MISMATCH
6. validate current user status
7. load current roles
8. resolve current permissions/menu_schema
9. update `last_seen_at`
10. return `menu_schema`

### Slice 1F — Logout Endpoint
Implement POST `/api/v1/auth/logout`.

Flow:
1. validate token/session
2. set `revoked_at`
3. set `revoked_reason = USER_LOGOUT`
4. audit LOGOUT / SESSION_REVOKED
5. return `local_clear_required = true`

## Role / Permission MVP Strategy
For implementation slice, avoid complex permission registry if not yet defined.

MVP option:
- create static API-side permission/menu mapping in a dedicated constants module
- do not put role-menu logic in GAS
- keep mapping small and explicit
- defer database permission registry to later

Required:
- roles are additive
- DIRECTOR is not automatically all permissions
- duplicates removed server-side
- `menu_schema` generated by API

## Environment / Configuration Requirements
Required environment values:
- Supabase URL
- Supabase service role key or secure server-side DB access method
- auth token hash secret if HMAC_SHA256 is selected
- API version
- session TTL policy

Forbidden:
- expose service role key to GAS
- store secrets in repo
- log secrets

## Hashing / Token Strategy Decision Needed
Password hashing:
- ARGON2ID or BCRYPT according to stored `password_algorithm`.
- If current users do not yet have `password_hash`, login cannot be tested until admin provisioning slice exists.

Session token hashing:
Recommended:
- HMAC_SHA256 with server secret
or
- SHA256 if no secret-based HMAC is available

Gemini previously allowed SHA256/HMAC_SHA256.
Implementation should prefer HMAC_SHA256 if env secret is available.

## Testing / Smoke Strategy
Before code:
- identify current test framework
- avoid hitting production accidentally
- create controlled local/unit tests if possible
- use fake passwords/tokens only
- no real secrets in tests

Smoke tests after implementation:
1. login with missing fields returns validation error
2. invalid credentials returns AUTH_INVALID_CREDENTIALS
3. missing terminal returns AUTH_TERMINAL_REQUIRED
4. refresh_menu with invalid token returns AUTH_SESSION_INVALID
5. logout with invalid token returns AUTH_SESSION_INVALID

Full successful login test requires:
- admin-provisioned test user
- password_hash
- assigned terminal
- roles
This should be planned separately.
