# EDS Power Global Rules

## 1. Призначення

Цей файл описує глобальні правила роботи системи EDS Power.

Правила з цього файлу діють для:

- всіх модулів
- всіх API-запитів
- всієї документації
- всієї роботи з AI-інструментами

---

## 2. Джерело правди

Репозиторій є єдиним джерелом правди.

Головні системні файли:

- `docs/00_SYSTEM/00_SYSTEM_OVERVIEW.md`
- `docs/00_SYSTEM/02_GLOBAL_RULES.md`
- `docs/00_SYSTEM/03_ARCHITECTURE.md`
- `docs/00_SYSTEM/04_DATA_CONTRACTS.md`
- `docs/00_SYSTEM/11_IDEA_NORMALIZER_RULES.md`
- `docs/00_SYSTEM/12_IDEA_MASTER_LOG.md`

**За замовчуванням** репозиторій і документація узгоджуються: якщо **реалізація суперечить задокументованому контракту** у звичайному циклі розробки — спочатку оновлюється **документація**, потім **код**, щоб знову відповідати узгодженому контракту.

**Виняток (objective truth — без «захисту застарілого тексту»):** якщо **верифікована реалізація** відображає **об’єктивну коректність**, **безпеку**, **цілісність персистентності** або **фактичну істинність системи**, а документація **застаріла або хибна**, **документація оновлюється негайно**, щоб відповідати **перевіреній** реальності. **Заборонено** «вирівнювати» код під **хибний** специфікаційний текст. Мета: governance **не захищає неправду**.

## 3. Порядок внесення змін

Будь-яка зміна проходить шлях:

```text
Ідея → TASK → аналіз → погодження → реалізація → оновлення документації → commit → push
```

Заборонено:

- реалізовувати ідею без TASK
- змінювати архітектуру без опису
- додавати нову логіку без оновлення документації

### Idea Normalizer Rule

All new ideas must pass Idea Normalizer before tasking.

No raw user idea may directly alter the repository, API, GAS, product logic, modules, or roadmap.

Idea Normalizer rules are defined in:

`docs/00_SYSTEM/11_IDEA_NORMALIZER_RULES.md`

The normalized idea log is maintained in:

`docs/00_SYSTEM/12_IDEA_MASTER_LOG.md`

## 4. Правила модулів

Кожен модуль:

- має власну зону відповідальності
- має окрему папку в `docs/`
- має описані входи
- має описані виходи
- має описані зв’язки з іншими модулями
- не дублює логіку інших модулів

Модуль не має права напряму керувати іншим модулем.

Взаємодія між модулями відбувається тільки через API та описані контракти.

### AUTH MVP freeze rule

AUTH module is frozen at MVP scope until `00-02_CALC_CONFIGURATOR` completes one full end-to-end scenario.

AUTH may include only:

- user identity
- basic session/auth check
- role placeholder
- avatar reference if needed

AUTH must NOT expand into full RBAC before CALC MVP works.

## 5. Правила API

API є центральним рівнем бізнес-логіки.

API:

- приймає JSON-запити
- перевіряє дані
- визначає модуль
- виконує дію
- повертає JSON-відповідь

Один запит = одна дія.

Заборонено:

- виконувати декілька незалежних процесів одним запитом
- змінювати декілька модулів одночасно
- передавати неструктуровані масиви
- змінювати формат JSON без оновлення `04_DATA_CONTRACTS.md`

---

## 6. Правила GAS

GAS є тонким клієнтом.

GAS може:

- читати дані з Google Sheets
- показувати меню та модальні вікна
- формувати JSON-запит
- відправляти запит в API
- отримувати відповідь
- показувати результат користувачу

GAS не може:

- виконувати бізнес-логіку
- виконувати розрахунки
- самостійно змінювати структуру даних
- приймати архітектурні рішення

### Дозволена client-side логіка

UI / GAS може виконувати тільки первинну перевірку введення перед відправкою запиту в API.

Дозволено:

- перевіряти, що обов’язкові поля заповнені
- перевіряти простий формат введення
- показувати користувачу локальні повідомлення
- не відправляти явно порожній або некоректний запит

Заборонено:

- виконувати бізнес-логіку
- виконувати розрахунки
- визначати модульний алгоритм
- змінювати структуру payload
- приймати рішення, які належать API

### Validation Responsibility Layers

Валідація розділяється між UI / GAS та API.

UI / GAS відповідає тільки за первинну перевірку форми:

- наявність обов’язкових полів
- простий формат введення
- локальні повідомлення користувачу
- блокування явно порожнього запиту

API відповідає за системну перевірку:

- відповідність data contracts
- права доступу
- бізнес-правила модуля
- нормалізацію даних
- вибір алгоритму
- фінальне рішення про виконання або відхилення дії

Правило:

- UI / GAS не може замінювати API validation
- API validation є фінальним джерелом істини для виконання дії

### Draft vs Validation Save Rule

DRAFT:

- may contain incomplete data
- may contain validation warnings
- may be saved for iterative work
- must still preserve structural JSON integrity

VALIDATED:

- all required fields complete
- all module validation passed
- safe for downstream use

LOCKED:

- immutable except through approved revision/change flow

RULE:

System must not block DRAFT saves for incomplete business data, but must block structurally invalid payloads.

Examples:

- Missing `voltage_class` → allowed in DRAFT, blocked in VALIDATED
- Broken JSON schema → blocked always

## 7. Правила бази даних

Supabase є рівнем збереження даних.

Database:

- зберігає дані
- зберігає статуси
- зберігає історію
- надає дані для API

Database не повинна:

- містити бізнес-логіку модуля
- виконувати розрахунки
- напряму взаємодіяти з користувачем

---

## 8. Правила файлів

Файли не зберігаються напряму в таблицях БД.

Файли зберігаються у файловому сховищі:

- Google Drive
- Supabase Storage
- інше сховище, якщо буде описано окремо

У БД зберігаються:

- file_id
- storage_provider
- file_path або file_url
- file_type
- version
- status
- зв’язок з модулем / об’єктом / користувачем

## 10. Правила AI-інструментів

AI не є джерелом правди.

AI може:

- аналізувати документацію
- пропонувати рішення
- писати код
- перевіряти логіку
- шукати конфлікти

AI не може:

- самостійно змінювати архітектуру
- вигадувати нові модулі без TASK
- змінювати контракти без погодження

## 11. Правила документації

Кожен файл має описувати тільки одну тему.

Заборонено:

- змішувати архітектуру, код, UI і бізнес-логіку в одному файлі
- дублювати одні й ті самі правила в багатьох місцях
- залишати застарілі правила без позначки

## Naming Convention Rules

### Module folders

Module folders use format:

`00-XX_MODULE_NAME`

Example:

`00-01_AUTH`
`00-02_CALC_CONFIGURATOR`

### Files inside module folders

Files inside every module use fixed numbering:

`00_MODULE_OVERVIEW.md`
`01_GOAL.md`
`02_INPUTS.md`
`03_PROCESS_FLOW.md`
`04_OUTPUTS.md`
`05_FUNCTIONS.md`
`06_DB_TABLES.md`
`07_MODULE_LINKS.md`
`08_UI_UX.md`
`09_STATUS.md`
`10_CONFIG.md`

### Product submodule folders

Product submodules use format:

`NN_PRODUCT_CODE`

Example:

`09_KZO`

### Files inside product submodules

Files inside every product submodule use fixed numbering:

`00_MVP_SCOPE.md`
`01_GOAL.md`
`02_INPUTS.md`
`03_ALGORITHM.md`
`04_OUTPUTS.md`
`05_FUNCTIONS.md`
`06_DATA_MODEL.md`
`07_VALIDATION.md`
`08_STATUS.md`
`09_AUDIT.md`

### Strict rule

One number = one role.

Duplicate numeric prefixes inside one folder are forbidden.

Cursor must not create a new numbered file if this number already exists.

If a new documentation type is needed, Cursor must:

1. create a TASK first;
2. explain why existing files are not enough;
3. propose the new number;
4. wait for approval.

## 16. Правила статусів модулів і системних документів

Для всіх модулів, TASK і системних документів використовується єдина система статусів:

- planned — ідея або заплановано
- draft_ready — структура описана, перевірена, готова до реалізації
- approved — затверджено як правильне рішення
- in_development — реалізується в коді
- active — працює в системі
- deprecated — застаріло, не використовується

Правила:

- Новий модуль починається зі status: planned
- Після заповнення структури та перевірки → draft_ready
- Після підтвердження архітектури → approved
- Під час реалізації → in_development
- Після запуску → active
- Старі або замінені рішення → deprecated

Заборонено:

- використовувати довільні статуси
- змінювати статус без оновлення документації
- мати модуль без STATUS.md

## 17. Правила CHANGELOG та щоденного аналізу

Система EDS Power ведеться через обов’язковий `CHANGELOG.md`.

### Призначення CHANGELOG:

- фіксація етапів розвитку
- контроль продуктивності
- аналіз план / факт
- управління ризиками
- підготовка презентацій
- історія ключових рішень

### Кожен етап у CHANGELOG повинен містити:

- Ціль
- План
- Факт
- Досягнення
- Продуктивність
- Ключовий прорив
- Ризики
- Статус

---

### Щоденне правило

На початку кожного нового робочого дня виконується обов’язковий аналіз попереднього дня.

AI (Cursor / GPT / інший агент) повинен:

- сформувати короткий звіт за попередній день
- показати:
  - що планувалось
  - що виконано
  - що не завершено
  - ключові прориви
  - ключові помилки / ризики
- підготувати матеріал для оновлення `CHANGELOG.md`
- допомогти визначити фокус поточного дня

---

### Формат щоденного звіту

- Вчорашня ціль
- Виконано
- Не виконано
- Проблеми
- Прорив
- Рекомендація на сьогодні

---

### Правила

- CHANGELOG оновлюється тільки після аналізу
- NOW відображає тільки поточний стан

## 18. Global Business Object Lifecycle

Для бізнес-об’єктів системи використовується єдиний життєвий цикл.

Базові етапи:

- created — об’єкт створено
- validated — об’єкт перевірено
- processed — об’єкт обробляється або оброблений модулем
- stored — результат збережено
- returned — статус або результат повернуто користувачу

Базові статуси:

- draft — об’єкт створено, але ще не підтверджено
- validation_error — об’єкт не пройшов перевірку
- ready — об’єкт готовий до наступної дії
- in_progress — об’єкт знаходиться в процесі виконання
- completed — об’єкт завершено
- cancelled — об’єкт скасовано
- archived — об’єкт більше не активний, але зберігається в історії

Правила:

- статус бізнес-об’єкта змінюється тільки через API
- статус має відповідати поточному етапу процесу
- модуль не може вигадувати власні статуси без опису в документації
- модульні статуси можуть деталізувати глобальні тільки в межах свого модуля
- історія зміни статусів має бути доступна для аудиту, якщо це потрібно модулю

Заборонено:

- змінювати статус напряму з UI / GAS
- використовувати неописані статуси
- змішувати статус системного документа зі статусом бізнес-об’єкта

## 19. Governance Audit Budget Control

Мета: зупинити governance-loop і перейти до bounded implementation після завершення `8B.2D`.

Обов'язкові правила:

1. **Master audits заборонені за замовчуванням**; дозволені лише за явною командою користувача.
2. **Post-fix audits заборонені** після вердикту `PASS` або `PASS WITH MINOR DOC FIXES`.
3. Для одного doctrine dossier дозволяється **рівно один** focused Gemini audit.
4. Якщо focused audit дає `PASS` або `PASS WITH DOC FIXES`, застосовується doc-fix pack і слайс **закривається** без повторних audit-циклів.
5. Реєстрові правки виконуються **лише** коли змінюється canonical state (без churn).
6. `8B.2D` є **останнім governance-only doctrine** перед bounded implementation slice.
7. `8B.2E` не відкривається, доки користувач явно не схвалить його як blocking prerequisite.

Операційний принцип:

- Governance служить переходу до продуктної цінності, а не нескінченним audit-петлям.
