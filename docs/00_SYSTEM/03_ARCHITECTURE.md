# EDS Power Architecture

## 1. Базовий принцип архітектури

Система EDS Power побудована як послідовна взаємодія рівнів:

```text
UI (Google Sheets / Web)
↓
GAS (тонкий клієнт)
↓
API (Render)
↓
Database (Supabase)

Цей ланцюг є базовим для системи і відповідає docs/00_SYSTEM/00_SYSTEM_OVERVIEW.md.

2. Ролі рівнів
2.1. UI (Google Sheets / Web)

UI є рівнем взаємодії користувача із системою.

Роль UI:

приймає введення даних від користувача
ініціює дії системи
показує користувачу результат або статус виконання
не містить бізнес-логіки
2.2. GAS

GAS є тонким клієнтом між UI та API.

Роль GAS:

передає дані з UI до API
отримує відповідь від API
повертає результат або статус на рівень UI
не виконує бізнес-логіку
не змінює структуру даних
2.3. API (Render)

API є центральним рівнем обробки дій системи.

Роль API:

приймає запити від GAS
виконує перевірку даних
нормалізує дані
визначає модуль
обирає потрібний алгоритм
виконує розрахунок або іншу дію модуля
зберігає результат у базі даних
формує відповідь
2.4. Database (Supabase)

Database є рівнем збереження даних системи.

Роль Database:

зберігає дані системи
надає дані для API
не містить бізнес-логіки
не виконує розрахунки
не взаємодіє з користувачем
3. End-to-end flow одного запиту

Один запит у системі проходить такий шлях:

Користувач вводить дані в UI
UI ініціює дію через GAS
GAS формує запит і надсилає його до API
API приймає запит
API виконує перевірку даних

5.1. Якщо перевірка не пройдена:

API повертає помилку
подальше виконання не відбувається
API нормалізує дані
API визначає модуль
API обирає алгоритм
API виконує дію (розрахунок або іншу логіку)
API зберігає результат у Database
Database повертає результат операції
API формує відповідь
GAS отримує відповідь
UI відображає результат користувачу
4. Архітектурні правила

Кожен модуль системи:

має власну зону відповідальності
має описані входи та виходи
працює тільки через API
не порушує інші модулі

Додаткові правила:

репозиторій є єдиним джерелом правди
будь-яка зміна спочатку описується, потім реалізується
код і документація повинні бути синхронні
5. Формат передачі даних

Взаємодія між рівнями відбувається через JSON.

Основні принципи:
всі запити мають єдиний формат
всі відповіді мають єдиний формат
структура не змінюється без оновлення документації
Приклад запиту
{
  "module": "CALC_CONFIGURATOR",
  "action": "calculate",
  "payload": {
    "product_type": "KSO",
    "parameters": {}
  }
}
Приклад відповіді
{
  "status": "success",
  "data": {},
  "error": null
}
Можливі значення status:
success — виконано успішно
error — помилка виконання
validation_error — помилка перевірки даних
6. Модульна логіка в API

API є єдиною точкою входу в систему.

Кожен запит містить:

module — назва модуля
action — дія
payload — дані

API:

визначає модуль
передає виконання відповідному обробнику
Принцип:
один запит = одна дія

Запит не повинен:

виконувати декілька незалежних процесів
змінювати декілька модулів одночасно
7. Обмеження GAS

GAS не містить бізнес-логіки.

GAS:

не виконує розрахунки
не приймає архітектурні рішення
не змінює структуру даних

GAS виконує тільки:

передачу запиту
отримання відповіді
відображення результату
