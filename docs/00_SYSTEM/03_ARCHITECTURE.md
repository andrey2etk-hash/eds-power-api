# EDS Power Architecture

## 1. Базовий принцип архітектури

Система EDS Power побудована як послідовна взаємодія рівнів:

```text
UI (Google Sheets / Web)
↓
GAS (тонкий клієнт)
↓
API (Render)
↓
Database (Supabase)
```

Цей ланцюг є базовим для системи і відповідає docs/00_SYSTEM/00_SYSTEM_OVERVIEW.md.

## 2. Ролі рівнів

### 2.1. UI (Google Sheets / Web)

UI є рівнем взаємодії користувача із системою.

Роль UI:

- приймає введення даних від користувача
- ініціює дії системи
- показує користувачу результат або статус виконання
- не містить бізнес-логіки

### 2.2. GAS

GAS є тонким клієнтом між UI та API.

Роль GAS:

- передає дані з UI до API
- отримує відповідь від API
- повертає результат або статус на рівень UI
- не виконує бізнес-логіку
- не змінює структуру даних

### 2.3. API (Render)

API є центральним рівнем обробки дій системи.

Роль API:

- приймає запити від GAS
- виконує перевірку даних
- нормалізує дані
- визначає модуль
- обирає потрібний алгоритм
- виконує розрахунок або іншу дію модуля
- зберігає результат у базі даних
- формує відповідь

### 2.4. Database (Supabase)

Database є рівнем збереження даних системи.

Роль Database:

- зберігає дані системи
- надає дані для API
- не містить бізнес-логіки
- не виконує розрахунки
- не взаємодіє з користувачем

## 3. End-to-end flow одного запиту

Один запит у системі проходить такий шлях:

1. Користувач вводить дані в UI
2. UI ініціює дію через GAS
3. GAS формує запит і надсилає його до API
4. API приймає запит
5. API виконує перевірку даних

5.1. Якщо перевірка не пройдена:

- API повертає помилку
- подальше виконання не відбувається

6. API нормалізує дані
7. API визначає модуль
8. API обирає алгоритм
9. API виконує дію (розрахунок або іншу логіку)
10. API зберігає результат у Database
11. Database повертає результат операції
12. API формує відповідь
13. GAS отримує відповідь
14. UI відображає результат користувачу

## 4. Архітектурні правила

Кожен модуль системи:

- має власну зону відповідальності
- має описані входи та виходи
- працює тільки через API
- не порушує інші модулі

Додаткові правила:

- репозиторій є єдиним джерелом правди
- будь-яка зміна спочатку описується, потім реалізується
- код і документація повинні бути синхронні

## 5. Формат передачі даних

Взаємодія між рівнями відбувається через JSON.

Основні принципи:

- всі запити мають єдиний формат
- всі відповіді мають єдиний формат
- структура не змінюється без оновлення документації

Приклад запиту

```json
{
  "module": "CALC_CONFIGURATOR",
  "action": "calculate",
  "payload": {
    "product_type": "KSO",
    "parameters": {}
  }
}
```

Приклад відповіді

```json
{
  "status": "success",
  "data": {},
  "error": null
}
```

Можливі значення status описані в docs/00_SYSTEM/04_DATA_CONTRACTS.md.

## 6. Модульна логіка в API

API є єдиною точкою входу в систему.

Кожен запит містить:

- module — назва модуля
- action — дія
- payload — дані

API:

- визначає модуль
- передає виконання відповідному обробнику

Принцип:

- один запит = одна дія

Запит не повинен:

- виконувати декілька незалежних процесів
- змінювати декілька модулів одночасно

## 7. Обмеження GAS

GAS не містить бізнес-логіки.

GAS:

- не виконує розрахунки
- не приймає архітектурні рішення
- не змінює структуру даних

GAS виконує тільки:

- передачу запиту
- отримання відповіді
- відображення результату

## 8. Future Async Pattern note

Поточна архітектура працює як синхронний flow:

```text
UI → GAS → API → Database → API → GAS → UI
```

У майбутньому для довгих операцій може бути описана асинхронна схема.

Ця примітка:

- не змінює поточну архітектуру
- не додає нових компонентів
- не дозволяє реалізацію async без окремого TASK
- фіксує тільки майбутню можливість для масштабування

## 9. Transaction Safety principle

Будь-яка дія, яка змінює дані, має виконуватись без частково збереженого неконсистентного результату.

Принцип:

- API відповідає за цілісність операції
- результат зберігається тільки після успішної перевірки даних
- помилка має повертати статус без часткового виконання бізнес-дії
- модуль не повинен змінювати дані іншого модуля напряму
- якщо дія потребує кількох кроків, порядок і умови збереження мають бути описані в документації модуля

Це правило не додає нових компонентів і не змінює поточну архітектуру.
